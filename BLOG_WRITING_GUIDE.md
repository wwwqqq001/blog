# Otaku57 博客写作规范

这份规范用于后续新增或重写项目文章。博客定位不是新闻站，也不是 README 镜像，而是“项目作品集 + 工程复盘”。每篇文章要让读者快速判断：这个项目是什么、解决了什么、我做了什么、有什么可看的结果。

## 参考来源

- [Google Developer Documentation Style Guide](https://developers.google.com/style)：优先保证技术内容清晰、一致，并以项目自身规范为最高优先级。
- [Microsoft Style Guide - Developer content](https://learn.microsoft.com/en-us/style-guide/developer-content/)：读者默认具备基本开发知识，文章要跳过泛泛科普，聚焦技术特定信息和具体目标。
- [GitLab Documentation Style Guide](https://docs.gitlab.com/development/documentation/styleguide/)：内容要简洁、直接、精确，方便搜索和扫读。
- [How to write a great technical blog post](https://medium.com/free-code-camp/how-to-write-a-great-technical-blog-post-414c414b67f6)：技术文章需要明确目标、受众和结构，并在发布前处理标题、图片和呈现细节。

## 总体原则

- 先说结论，再讲过程。首屏必须回答“这是什么项目”和“为什么值得放进作品集”。
- 用事实支撑评价。优先写仓库、部署地址、技术栈、运行方式、截图、架构图、已知状态；少写空泛形容。
- 区分原创、Fork、部署实践、归档项目。不能把 fork 或占位页写成从零原创项目。
- 每篇只讲一个主线。不要在同一篇里同时讲项目介绍、教程、部署日志、未来规划和个人感想。
- 公开边界必须明确。涉及私有仓库、Cookie、Token、Webhook、账号、Cloudflare 配置时，只写结构和原则，不写真实值。
- 不为了凑字数扩写。项目内容少就写短文，状态不完整就写“状态记录”，不要编造功能。

## 文章类型

### 1. 原创项目复盘

适用：自己主要实现的工具、模组、脚本、前端页面。

固定结构：

```md
# 项目名：一句话定位

创建时间：YYYY-MM-DD HH:mm UTC
仓库：<https://github.com/...>
线上地址：<https://...>（如有）
状态：公开仓库 / 私有仓库 / 运行中 / 已归档

一段 80-140 字摘要：说明项目解决什么问题、为什么要做、当前做到什么程度。

![项目截图或架构图](路径)

## 问题背景

## 我的实现

## 关键取舍

## 运行结果

## 后续计划

## 展示边界
```

写法要求：

- “我的实现”要写具体模块、流程或技术点，不写“我做了优化”这种空话。
- “关键取舍”至少写一个限制或妥协，例如运行时限制、部署成本、兼容性、安全边界。
- “运行结果”优先用截图、命令输出摘要、页面状态、部署状态或实际效果说明。

### 2. 自动化脚本 / 定时任务

适用：签到、通知、GitHub Actions、Cloudflare Cron、Workers 定时任务。

固定结构：

```md
# 项目名：把某个重复动作自动化

创建时间：
仓库：
运行位置：GitHub Actions / Cloudflare Workers / Docker / 本地
状态：

## 任务目标

## 执行流程

## 配置与安全边界

## 失败处理

## 适合继续改进的地方
```

写法要求：

- 必须说明触发方式、凭据放在哪里、结果如何通知。
- 不公开真实 Cookie、Token、Webhook、账号信息。
- 如果有截图，优先展示流程图、Actions 状态页、公开 UI，不展示敏感日志。

### 3. Cloudflare Pages / Workers 部署项目

适用：已经部署到 Cloudflare 的网页、入口页、Worker 服务。

固定结构：

```md
# 项目名：Cloudflare 上的什么入口

Cloudflare 地址：<https://...>
自定义域名：<https://...>（如有）
状态：正在运行 / 返回 404 / 占位页 / 已归档
来源：Wrangler 项目列表，最近更新约 X

一段摘要：说明当前公开能看到什么，不夸大未完成内容。

![公开页面截图](路径)

## 当前状态

## 为什么放进博客

## 后续处理

## 展示边界
```

写法要求：

- 正常站点写“当前能做什么”；占位页写“链路已打通但内容未替换”；404 项目写“部署历史/待清理”。
- 不要把 nginx 默认页、404 页写成完整产品。
- 截图必须来自公开页面，或使用明确标注的状态示意图。

### 4. Fork / 自托管实践

适用：从别人项目 fork、部署、研究或改造。

固定结构：

```md
# 项目名：一次某方向实践

创建时间：
上游 / 仓库：
状态：公开 fork / 私有 fork / 自托管实践

## 为什么关注它

## 我实际做了什么

## 架构或能力拆解

## 可改造方向

## 展示说明
```

写法要求：

- 开头必须说明这是 fork 或自托管实践。
- “我实际做了什么”不能省略：部署、配置、阅读源码、改 UI、接存储、排错，都要写清楚。
- 不把上游功能包装成自己的原创成果。

### 5. 归档 / 待清理项目

适用：早期实验、404 页面、已废弃仓库、只保留部署记录的项目。

固定结构：

```md
# 项目名：早期部署记录

地址：
状态：
来源：

## 当前状态

## 保留原因

## 后续处理建议

## 展示边界
```

写法要求：

- 短文即可，300-700 字足够。
- 重点是资产清单和状态说明，不写不存在的功能。

## 标题规范

- 格式：`项目名：一句话价值`
- 标题不要超过 28 个汉字。
- 标题里的价值要具体，例如：
  - 好：`GitHub Cron：运行在 Cloudflare Pages 的 Actions 触发器`
  - 差：`GitHub Cron：一个很有意思的小工具`
- 对归档项目直接写状态，不美化：
  - 好：`asdqe：早期 Cloudflare Pages 部署记录`
  - 差：`asdqe：一个神秘的云端项目`

## 摘要规范

摘要用于列表页，长度控制在 50-90 个汉字。必须包含：

- 项目类型
- 核心用途
- 当前状态或关键技术

示例：

```text
运行在 Cloudflare Pages 上的轻量入口，用于把 GitHub Actions 触发流程整理成可访问的管理页面。
```

不要写：

```text
这是一个非常好用、非常强大、非常值得记录的小项目。
```

## 段落和语气

- 每段 2-4 行，避免长段。
- 多用短句，少用绕口解释。
- 可以用第一人称，但要克制。优先写“我做了什么”和“我为什么这样取舍”，少写情绪。
- 不写营销词：极致、强大、丝滑、优雅、革命性、完美、全能。
- 不写 AI 味模板句：不是 X，而是 Y；这不仅是 X，更是 Y；在数字化时代。
- 术语保持一致：Cloudflare Workers、Cloudflare Pages、GitHub Actions、Secrets、Cron、R2、D1。

## 图片规范

每篇至少一张图，除非是短归档记录。

优先级：

- 公开页面截图
- 架构图或流程图
- 状态示意图
- 仓库结构图

图片说明必须写清楚：

- 图片来自哪里
- 展示了什么
- 哪些内容没有展示

不要使用：

- 暗色模糊背景图
- 和项目无关的装饰图
- 暴露密钥、账号、日志、后台配置的截图

## 安全与真实性规则

- 不写真实 Token、Cookie、Webhook、邮箱登录态、账号 ID。
- 私有仓库只写“私有仓库”，不写不可公开的源码细节。
- 页面当前是 404 就写 404；当前是 nginx 默认页就写占位页。
- 不确定的信息标注来源，例如“Wrangler 项目列表显示最近更新约 3 天前”。
- 编造内容只能用于非关键叙述，不能编造功能、性能、用户量、上线时间、部署状态。

## 发布前检查

- 标题是否能说明项目价值。
- 开头 150 字内是否说清“这是什么”。
- 是否标明仓库、线上地址、状态。
- 是否区分原创、Fork、部署实践、归档。
- 是否至少有一张有说明的图片。
- 是否写了我实际做的内容。
- 是否写了限制、取舍或后续计划。
- 是否没有泄露密钥、账号、日志和私有配置。
- 摘要是否适合列表页展示。
- 图片路径是否能在本地页面正常加载。

## 重写旧文章时的优先级

先重写这些问题最明显的文章：

- 只有部署占位但写得像完整项目的文章。
- 只复述 README，没有写自己实践过程的文章。
- 没有截图或图示的文章。
- 没有明确“原创 / Fork / 私有仓库 / 归档”状态的文章。

每次重写 2-3 篇即可，避免一次性改太多导致风格失控。
