# Rainyun：把网页签到封装成可复现的容器任务

创建时间：2026-06-25 13:23:07 UTC
仓库：<https://github.com/wwwqqq001/Rainyun>
运行位置：Docker / GitHub Actions / 本地浏览器环境
状态：私有仓库

Rainyun 是一个网页自动化签到项目。这个任务麻烦的地方不在“点一下签到”，而在它要经过真实页面、验证码和浏览器状态。只写一个请求脚本很容易失效，所以我把它做成容器任务，让同一套流程能在本地、服务器和 GitHub Actions 里重复运行。

![Rainyun 自动化签到工作流](/blogs/rainyun/workflow.svg)

图里展示的是一次任务的实际路径：容器拉起浏览器，Selenium 控制页面，ddddocr 处理验证码，脚本保存截图和日志，最后把结果发到通知渠道。这里刻意不展示账号、代理或 Webhook，只保留可以公开的运行结构。

## 任务目标

这个项目要解决的是“每天都要打开网页完成一段固定流程”的问题。网页任务天然不稳定，真正需要处理的是这些现场问题：

- 页面加载速度会变。
- 弹窗和验证码会打断流程。
- 浏览器版本会影响元素选择。
- 脚本失败后只看日志很难判断现场。

所以 Rainyun 的目标不是写一个最短脚本，而是把签到做成一次可观察、可复现、可排错的任务。

## 我做了什么

核心脚本用 Selenium 驱动浏览器完成页面操作。验证码相关逻辑交给 ddddocr，减少人工介入。运行层用 Docker 固定浏览器和依赖，避免“本地能跑，云端不能跑”的情况。

仓库里几个关键部分分别承担不同职责：

- `rainyun.py`：页面打开、元素定位、验证码处理、签到执行。
- `Dockerfile`：固定运行环境和浏览器依赖。
- `docker-compose.yml`：给本地或服务器提供稳定启动入口。
- `.env.example`：说明需要哪些配置，但不保存真实值。
- `script/`：放置辅助运行脚本。

## 为什么这样做

我没有把它写成纯接口请求脚本。纯接口方式更轻，但对页面变化和验证流程更敏感，也更容易在接口变更后直接失效。Selenium 更重，却能更贴近真实用户路径。

容器化也是同样的取舍。它增加了镜像构建和调试成本，但换来的是运行环境一致：浏览器版本、字体、依赖和启动方式都能被固定下来。

## 运行后能看到什么

一次完整运行至少会留下三类结果：

- 签到是否成功。
- 失败时停在哪个页面。
- 通知渠道是否把结果发出去。

其中截图留证最重要。网页自动化失败时，“元素没找到”这类日志只说明结果，不说明原因；截图能直接看出是验证码、弹窗、登录态还是页面改版导致的问题。

## 后续计划

后续更适合按职责拆分：

- 页面操作层：只处理浏览器和元素选择。
- 验证码层：只处理识别和失败回退。
- 调度层：只处理多账号和执行顺序。
- 通知层：只处理不同推送渠道。
- 报告层：统一整理截图、日志和结果。

这样页面结构变化时，主要改页面操作层，不会牵动整条任务链路。
