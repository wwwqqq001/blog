# Rainyun：容器化的雨云自动签到

创建时间：2026-06-25 13:23:07 UTC  
仓库：<https://github.com/wwwqqq001/Rainyun>  
状态：私有仓库

Rainyun 是一个更偏“工程化”的自动签到项目。

它不是单纯请求接口，而是用 Selenium 模拟浏览器操作，再配合 ddddocr 处理验证码识别，把一个容易受页面变化影响的流程封装到 Docker 和 GitHub Actions 里。

![Rainyun 自动化签到工作流](/blogs/rainyun/workflow.svg)

这张图更偏运行现场：容器启动后拉起浏览器，Selenium 驱动页面，验证码识别模块参与决策，最后把截图、日志和通知组合成一次完整报告。它展示的是“为什么这个项目需要 Docker”，因为浏览器自动化的稳定性很大程度取决于运行环境是否可复现。

我也把截图放在链路里，是为了突出失败排查的价值。页面自动化不像纯 API 脚本，失败时日志往往只能告诉你“没找到元素”，截图才能说明页面到底停在了验证码、登录页、弹窗还是结果页。

## 问题背景

雨云签到有验证码和页面流程，直接请求不稳定，也不太适合把逻辑写成一个简单 API 调用。

于是这个项目的思路变成：

- 像真人一样打开页面。
- 用浏览器自动化完成操作。
- 遇到验证码时尝试识别。
- 成功或失败都截图。
- 最后把结果推送出去。

## 技术结构

仓库里比较关键的文件：

- `rainyun.py`：核心自动化逻辑。
- `Dockerfile`：容器化运行环境。
- `docker-compose.yml`：本地或服务器部署入口。
- `.env.example`：配置模板。
- `script/`：额外运行脚本。

这个项目选择 Docker 是因为浏览器自动化对环境依赖比较敏感。本地装得对，不代表服务器也跑得起来。容器能把浏览器、依赖和运行方式固定下来。

## 项目亮点

### 多账号支持

账号信息可以按分隔符配置，脚本统一调度。这个设计让任务从“给一个账号写脚本”变成“给一组账号跑流程”。

### 截图留证

自动化任务失败时，日志经常不够。截图能直接回答：页面变了、验证码没过、按钮没点到，还是接口返回了异常。

### 推送降级

通知内容可能很长，尤其带截图或详细报告时。这个项目把推送当作可靠性的一部分处理，而不是简单 `send()` 一下。

## 可以继续改进的地方

如果后续继续维护，我会把它拆得更清楚：

- 浏览器操作层。
- 验证码识别层。
- 任务调度层。
- 通知适配层。
- 运行报告层。

这样以后页面变化时，只需要改浏览器操作，不会牵动所有逻辑。
