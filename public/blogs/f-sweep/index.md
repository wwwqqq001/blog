# F-Sweep：给 Minecraft 容器界面补一个快速整理动作

创建时间：2026-02-28 01:09:14 UTC
仓库：<https://github.com/wwwqqq001/F-Sweep>
运行环境：Minecraft 1.21.1 / NeoForge
状态：私有仓库

F-Sweep 是一个 Minecraft NeoForge 模组。它解决的是一个很小但高频的体验问题：打开容器后，把可存入的物品快速转入 Beyond Dimensions 网络或相关存储，而不是一格一格手动移动。

![F-Sweep 容器交互流程图](/blogs/f-sweep/interaction.svg)

图里把一次操作拆成客户端和服务端两侧：按钮或快捷键只负责发起请求，槽位判断、物品转移和结果确认放在服务端完成，最后再把反馈显示到界面。这个拆法能避免“客户端看起来点了，服务端其实没执行”的问题。

## 问题背景

这类功能看起来只是“加一个按钮”，但 Minecraft 模组里真正要处理的是边界：

- 不是所有界面都应该出现按钮。
- 不是所有槽位都能被转移。
- 客户端看到的状态不一定等于服务端可执行的状态。
- 存储满了、没有可转移物品、目标不可用时都要反馈。

所以 F-Sweep 的重点不是按钮本身，而是一次安全、可预期的容器操作。

## 我做了什么

交互入口放在客户端：玩家打开容器后，可以点击按钮，也可以按快捷键 `F`。客户端不直接修改物品，只把请求发给服务端。

服务端负责真正的判断：

- 当前界面是否允许触发。
- 哪些槽位应该跳过。
- 目标存储是否可用。
- 转移完成后如何统计结果。

这种拆法让 UI 保持轻量，也避免客户端绕过服务端状态。

## 为什么不只做客户端按钮

我没有把所有逻辑放在客户端。客户端处理起来更直观，但会带来状态不一致和安全问题。服务端验证更啰嗦，却更符合多人游戏和模组交互的边界。

按钮和快捷键都保留，是为了照顾不同使用习惯：按钮适合发现功能，快捷键适合熟练使用。

## 玩家实际看到什么

最终体验应该足够短：

- 打开容器。
- 点击按钮或按 `F`。
- 可转移物品被处理。
- 界面给出成功、无物品或失败提示。

这类小模组如果没有反馈，玩家很难判断到底是没东西可转移，还是功能没有生效。因此提示文案也是功能的一部分。
