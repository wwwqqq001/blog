CloudPaste 是一个 Serverless 文件管理与 Markdown 分享工具。这个仓库是 fork，我把它放进作品集，是为了记录一次 Cloudflare 自托管工具研究：它怎样组织前端、Worker、D1、R2/S3 和 WebDAV，而不是把上游功能包装成自己的原创项目。

## 为什么关注它

我想要一个轻量文件工具，最好满足这些条件：

- 能管理文件。
- 能分享文本和 Markdown。
- 能接入不同存储。
- 能部署在 Cloudflare 或 Docker。
- 不需要长期维护传统服务器。

CloudPaste 的方向接近这个需求，所以它适合作为自托管实践记录。

## 我实际看了什么

这篇文章记录的是研究和部署形态梳理，而不是原创功能发布。我关注的动作包括：

- 阅读它如何拆分前端、API、元数据和对象存储。
- 梳理 D1、R2/S3、WebDAV 在系统里的位置。
- 评估 Cloudflare 环境下上传、预览和分享的限制。
- 判断它是否适合作为后续个人文件工具的基础。

把这些写清楚，比简单贴一个 fork 仓库链接更有价值。

## 架构或能力拆解

CloudPaste 有几个值得记录的能力：

- Cloudflare Workers 提供运行入口。
- D1 保存文件和分享相关元数据。
- R2 或 S3 兼容存储保存真实文件。
- WebDAV 让工具能接入更多客户端。
- Markdown 编辑器让文本分享不只是上传文件。
- 文件预览覆盖图片、视频、PDF、Office、代码等格式。

这些能力共同决定了它不是单纯网盘页面，而是一个 Serverless 文件管理工具。

## 我会优先改哪里

Cloudflare 很适合轻量服务，但文件工具会触碰一些边界：上传体积、执行时长、流量、预签名 URL、对象存储权限和预览性能都需要考虑。

如果后续基于它继续改造，我不会先追求功能数量，而会先处理个人部署体验：

- 配置更简单。
- 存储接入更清楚。
- 分享权限更安全。
- 移动端管理更顺手。
