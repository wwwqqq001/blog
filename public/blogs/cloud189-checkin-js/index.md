Cloud189CheckinJs 是一个轻量的签到自动化项目。它的目标很明确：把每天手动打开天翼云盘签到，改成到点自动执行，并在任务结束后给出结果。

## 任务目标

手动签到最大的问题不是一次操作麻烦，而是每天都要记得。自动化之后，也不能只考虑“请求发出去了没有”，还要处理这些边界：

- 凭据不能写进仓库。
- 运行失败要能被看到。
- 本地和 GitHub Actions 要共用一套核心逻辑。
- 推送渠道不能和业务逻辑绑死。

这个项目的目标是把这些小问题收拢成一条稳定任务链路。

## 我做了什么

项目主体使用 JavaScript。`package.json` 负责脚本入口和依赖，`.github` 工作流负责定时运行，核心签到逻辑放在源码目录中。通知渠道通过环境变量启用，不把真实配置写入仓库。

这种结构的好处是部署轻：本地可以直接跑，GitHub Actions 也能按计划执行；如果将来更换推送渠道，不需要重写签到逻辑。

## 凭据怎么放

这类脚本最容易出问题的是凭据处理。我的原则是：

- 仓库只保存配置说明，不保存真实账号。
- GitHub Actions 使用 Secrets 保存敏感值。
- 本地运行使用环境变量或本地配置文件。
- 博客只展示结构，不展示任何真实值。

自动化可以省事，但不能用泄露凭据换省事。

## 失败时怎么知道

签到任务如果没有反馈，很快就会变成黑盒。这个项目把通知当成任务的一部分，而不是附属功能。

运行结果需要覆盖三种状态：

- 成功：明确告诉我当天任务已完成。
- 失败：说明失败阶段，避免静默坏掉。
- 配置异常：提醒我检查凭据、推送渠道或 Actions 设置。

## 后续可以补什么

后续可以把任务状态记录得更结构化，例如保留最近几次运行结果、把失败原因分为网络错误、登录态失效、接口变更和推送失败。这样就不只是“自动签到”，而是一个可维护的个人自动化任务。
