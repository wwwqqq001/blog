# Cloud189CheckinJs：把天翼云盘签到变成自动流程

创建时间：2026-06-27 13:32:10 UTC  
仓库：<https://github.com/wwwqqq001/Cloud189CheckinJs>  
状态：私有仓库

这个项目的目标很直接：把天翼云盘每天要做的签到动作，从“记得打开页面”变成“到点自动跑完并推送结果”。

我把它放进作品集，不是因为它的业务有多复杂，而是因为它正好代表了一类很常见的个人自动化需求：逻辑不大，但涉及账号配置、运行环境、通知渠道、失败重试和安全边界。

## 为什么做

手动签到的问题不是麻烦一次，而是每天都要想起来。脚本化之后，真正要解决的也不是“发一个请求”这么简单，而是：

- 凭据不能直接写死在代码里。
- GitHub Actions、本地运行和其它环境要尽量共用一套逻辑。
- 推送渠道要可替换，不能绑定某一个服务。
- 脚本失败时要能看见，而不是静默坏掉。

## 技术选择

项目主体使用 JavaScript，运行方式偏轻：本地可以直接跑，云端可以交给 GitHub Actions 定时触发。

我更关心的是运行链路：

- `package.json` 管理脚本和依赖。
- `.github` 工作流负责定时执行。
- `src` 存放核心逻辑。
- 推送渠道通过环境变量启用。

## 设计重点

### 凭据隔离

签到脚本很容易犯的错，是把 `.env` 或真实账号提交到仓库。这个项目后续展示时只写配置方式，不展示具体值。

我希望它在博客里呈现的是一个原则：自动化可以省事，但不能用泄露凭据换省事。

### 多渠道通知

签到任务如果没有反馈，很快就会变成黑盒。所以通知不是附属功能，而是自动化任务的一部分。

这个项目保留了多种推送思路：企业微信、PushPlus、WxPusher、Telegram 等都可以按需要接入。

### 运行边界

这类脚本适合写清楚边界：

- 它只解决个人自动化。
- 它依赖第三方服务接口状态。
- 它不应该承诺永久稳定。
- 它不适合把账号信息公开托管。
