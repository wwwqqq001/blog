# 米游社自动签到：桌面扫码登录加云端多账号

创建时间：2026-06-25 起
仓库：<https://github.com/wwwqqq001/mihoyo-auto-checkin>
运行位置：本地 Python + Cloudflare Workers
状态：私有仓库，已接入统一调度平台

这个项目把“登录”和“每天签到”拆成两端。登录需要扫码和本机交互，适合桌面端；每天多账号签到适合放在 Cloudflare Worker。上线后，Worker 再作为调度平台的 Service Binding Handler，结果进入统一日报。

![米游社双端架构](/blogs/mihoyo-auto-checkin/architecture.svg)

## 快速了解

| 项目项 | 内容 |
| --- | --- |
| 类型 | 自动化脚本 / 双端系统 |
| 桌面端 | Python CLI / GUI，扫码登录、本地签到、导出凭证 |
| 云端 | TypeScript Worker，多账号管理、定时/平台触发、Web 面板 |
| 游戏覆盖 | 原神、绝区零等配置化游戏接口 |
| 存储 | Cloudflare KV 保存账号元数据 |

## 任务目标

- 用扫码登录减少手工复制 Cookie 的摩擦。
- 支持多账号启用、停用和重命名。
- 把游戏奖励整理成可展示、可回传的结构。
- 接入统一平台后，不再各自推送正式通知。

## 我做了什么

桌面端负责：

- 二维码获取与轮询。
- 登录态换票。
- 本地签到调试。
- 导出到 Worker。

Worker 负责：

- 账号列表与面板。
- 访问保护。
- 调用游戏签到接口。
- 向平台返回标准 payload。

请求层需要处理签名头和游戏差异。这块逻辑容易碎，所以我把它收敛成共享工具函数和游戏配置表，而不是每个游戏复制一套请求代码。

## 配置与安全边界

- 登录凭证和 Cookie 不进公开仓库。
- Worker 侧敏感配置使用 Secrets / 受控存储。
- 面板访问有密码保护。
- 博客只描述结构，不展示任何真实凭证。

## 失败处理

常见失败分几类：

- 登录态过期：需要重新扫码导出。
- 单游戏失败、其他游戏成功：账号结果里分开记录。
- 平台触发超时：由平台重试策略接管。

结构化 `accounts` 和 `roles` 字段的目的，就是让日报能说明“哪个账号的哪个角色拿到了什么”，而不是只显示一个总成功。
